# Omnilytic — Концептуальное описание (rebuild guide)

> Этот файл содержит смысл, логику и назначение приложения.
> Конкретных кодовых решений здесь нет — каждая нейросеть реализует их сама, исходя из описания.
> Указаны только те подходы, которые жёстко диктуются задачей (иного способа сделать то же самое просто нет).

---

## 1. Назначение приложения

Одностраничное веб-приложение для аналитики продавцов на Wildberries. Продавец загружает данные о продажах — либо файлом Excel (стандартный отчёт WB), либо запрашивает их напрямую из WB API по токену. Приложение показывает:

- **Ключевые метрики:** выручка, заказы, выкупы, конверсии, средний чек, процент отмен — с динамикой относительно предыдущего периода
- **Воронку продаж:** показы → корзина → заказы → выкупы (отмены)
- **Юнит-экономику:** структура затрат по одному товару — комиссия, логистика, хранение, эквайринг, штрафы, payout
- **Таблицы:** хиты продаж (топ по выручке), проблемные товары (нет заказов / низкий CTR / низкий выкуп)

Приложение не принимает решения за пользователя — оно только показывает цифры.

---

## 2. Архитектура (общая)

Бэкенд — HTTP API (REST). Фронтенд — SPA на любом стеке (vanilla JS, React, Vue, Svelte — неважно). Никаких жёстких требований к фреймворкам.

Единственное глобальное состояние хранится **в оперативной памяти бэкенда** (in-memory). При перезапуске данные теряются. База данных не используется — это сделано сознательно для упрощения.

Фронтенд при старте делает один запрос к health-endpoint, чтобы понять, загружены ли данные, и подтянуть список товаров. Дальше все запросы идут по мере взаимодействия пользователя.

## 3. Источники данных

### 3.1 Excel-файл

Стандартный отчёт Wildberries — `.xlsx` или `.xls` с русскими заголовками колонок (~47 колонок). Пользователь загружает файл через multipart-upload. Бэкенд читает файл, переименовывает колонки с русского на внутренние английские ключи (словарь маппинга), обогащает данные (динамика между периодами, классификация товаров, суммарный остаток, упущенная выручка).

После загрузки Excel период дат фиксирован — фильтр дат на фронтенде игнорируется, потому что Excel уже содержит данные за конкретный диапазон.

### 3.2 WB API

Пользователь вводит API-ключ (токен доступа к статистике Wildberries) и выбирает период (максимум 365 дней назад, не в будущем).

Бэкенд делает последовательные запросы:

1. **Sales Funnel** (`/api/v3/sales-funnel`) — основной источник: товары, их показы, заказы, выкупы, выручка, остатки. Пагинация курсором по 1000 записей
2. **Search Report** (`/api/v2/search-report`) — опционально, может отсутствовать в тарифе. Даёт показы и CTR. Если недоступен — soft-error, метрики помечаются

Данные маппятся в тот же внутренний формат, что и Excel (vendorCode → seller_article, nmId → wb_article). Обогащение — то же самое.

API-ключ сохраняется в localStorage браузера, чтобы при перезагрузке страницы не вводить заново.

### 3.3 Внутренний формат (processed_data)

Это основная структура — единый список словарей, в который приводятся данные из любого источника.

Каждый товар — один словарь с полями:
- seller_article (артикул продавца, строка)
- wb_article (артикул WB, строка или число)
- name (название товара)
- category (категория/предмет)
- brand (бренд)
- orders_qty, orders_value (заказы, шт и ₽) — и prev для предыдущего периода
- purchased_qty, purchased_value (выкупы)
- cancelled_qty, cancelled_value (отмены)
- impressions, card_views, add_to_cart (показы, переходы, корзина)
- stock_wb, stock_mp, total_stock (остатки)
- ctr, purchase_rate, order_conversion, cart_conversion (проценты)
- avg_price (средняя цена)
- card_rating, review_rating (рейтинги)
- orders_dynamics_pct, revenue_dynamics_pct (динамика в %)
- bcg_category (классификация: star/question/cash_cow/dog)
- lost_revenue (упущенная выручка из-за отсутствия остатка)
- и другие

## 4. Система фильтрации (обязательный подход)

Это один из ключевых механизмов, и он должен быть реализован именно так, как описано, потому что иначе не получится добиться нужного UX.

### 4.1 Два уровня фильтров

**General (общие)** — в шапке страницы. Даты (два date input) + мультивыбор товаров (дропдаун с поиском).

**Local (для каждой секции)** — каждая секция на странице (KPI, воронка, таблицы, блоки юнит-экономики) может иметь свои собственные фильтры дат и товаров.

### 4.2 Каскадное наследование (propagation)

Когда пользователь меняет общие фильтры — они применяются ко всем секциям, у которых нет локального переопределения.

Когда пользователь меняет локальный фильтр в секции — эта секция отвязывается от общих (override = true). Появляется визуальная нотификация «Локальный фильтр переопределяет общий».

Когда пользователь нажимает «Сбросить» в локальном фильтре — override снимается, секция начинает наследовать общие настройки.

### 4.3 Single-mode (для юнит-экономики)

Блоки юнит-экономики не могут показывать «все товары» — только один. В дропдауне нет галочки «Все товары». Если товар не выбран — placeholder «Выберите один товар».

### 4.4 Что ещё важно

- Поиск по товарам в дропдауне — строковый поиск по артикулу (seller_article, wb_article) и названию
- Дропдаун должен уметь: открываться/закрываться, показывать выбранные элементы с галочкой, иметь кнопки «Сбросить» и «Применить»
- При клике вне дропдауна он должен закрываться

## 5. Страницы и модули

### 5.1 Командный центр

Собирает данные из 3–5 параллельных запросов к API (зависит от того, какие модули включены).

**KPI-панель:** два ряда карточек. Каждая карточка содержит:
- Название метрики
- Текущее значение (крупно)
- Динамика (процент изменения к предыдущему периоду) — зелёная ↑ или красная ↓
- Разница в абсолютных единицах
- Значение предыдущего периода («пред: X»)

Метрики: выручка, заказы, выкупы, процент выкупа, конверсия в корзину, конверсия в заказ, средний чек, отмены, процент отмен.

KPI считаются как средневзвешенные: сумма по выбранным товарам / количество товаров с данными.

**Воронка продаж:** три визуальные полосы (добавления в корзину → заказы → выкупы), ширина каждой пропорциональна объёму. Показывает динамику и предыдущий период.

**Конверсии:** три линейные шкалы-заливки с цветовой индикацией (зелёный/жёлтый/красный — в зависимости от значения).

**Хиты продаж:** таблица товаров, отсортированных по выручке (топ-N). Колонки: номер, артикул + название, выручка, динамика (badge), остаток (с выделением OOS и <50).

**Зоны внимания:** проблемные товары — три категории:
- Нет заказов (много показов/переходов, но 0 заказов)
- Низкий CTR (только для Excel, где есть CTR)
- Низкий выкуп (<30% при >10 заказах)

Каждая проблема — строка таблицы с рекомендацией.

**AI-анализ (опционально):** кнопка → бэкенд собирает сводку → отправляет нейросети → текст анализа.

### 5.2 Юнит-экономика

Два блока, каждый работает с одним товаром (single-mode).

**Блок 1: Шкалы + круговая диаграмма**

10 шкал, каждая — полоса, разделённая на два сегмента:
- Левый (start) — значение компонента на дату начала периода. Ширина = доля от выручки на начало периода
- Правый (end) — значение компонента на дату конца периода. Ширина = доля от выручки на конец периода

Компоненты:
1. Итоговая цена (retail_price) — сумма продаж
2. Комиссия WB
3. Логистика
4. Хранение
5. Эквайринг
6. Штрафы
7. Приёмка
8. Удержания
9. Доплаты
10. К перечислению продавцу (payout = retail_price − все остальные)

Сверху — две сводки с выручкой на начало и конец периода.

Круговая диаграмма — структура затрат за весь период (без payout, без retail_price). Каждый компонент — цветной сегмент. Если сегмент нулевой — не показывается.

**Блок 2: Таблица тарифов**

8 строк, каждая с двумя измерениями:
- На начало периода: Стандарт / Факт
- На конец периода: Стандарт / Факт
- Изменение: Стандарт / Факт

Строки: комиссия (%), логистика (₽), хранение (₽), эквайринг (%), штрафы (₽), приёмка (₽), удержания (₽), доплаты (₽).

Стандарт — справочная комиссия WB по категории товара + методу доставки (FBS/FBO). Факт — реальные числа из финансового отчёта.

**Как UE получает данные:** бэкенд делает POST-запрос к `/api/finance/v1/sales-reports/detailed` с полями: rrdId, nmId, vendorCode, title, subjectName, rrDate, saleDt, deliveryMethod, retailPriceWithDisc, ppvzSalesCommission, acquiringFee, rebillLogisticCost, paidStorage, penalty, deduction, additionalPayment, paidAcceptance.

Пагинация по rrdId (курсор), лимит 100000 строк.

**Проблема с датами (обязательно учесть):** финансовый отчёт группируется по точной дате (rrDate). Если на date_from не было операций — данные для start-сегмента = 0. Нужно: если на точную дату данных нет — искать ближайшую дату с данными. Для date_to, равного сегодня — идти назад (к более ранним датам).

**Resolve ID:** когда пользователь выбирает товар в дропдауне, его id — это либо seller_article (артикул продавца), либо wb_article (артикул WB). WB API работает с nmId (числовой артикул WB). Поэтому перед запросом к финансовому API нужно найти товар в локальном DATA_STORE и подменить id на wb_article (если есть). Если wb_article пустой — использовать seller_article как есть.

## 6. Обязательные UI-состояния

У каждого компонента должны быть следующие состояния (иначе пользователь будет видеть белый экран без объяснения):

**Командный центр:**
- Данных нет → пусто
- Данные есть, метрик нет → «Нет данных по выбранным товарам»
- Данные есть, метрики есть → рендер

**Юнит-экономика:**
- Null (не загружено) → «Данные ещё не загружены»
- Нет API-токена → «Нужен WB API токен»
- Не выбран товар → «Выберите один товар»
- Ошибка → текст ошибки
- Данные есть → шкалы/диаграмма/таблица
- Pie пустой → «Нет данных для диаграммы»
- Тарифы пустые → «Нет тарифов»

**Фильтры:**
- Нет товаров вообще → «Нет загруженных товаров»
- Single-mode, не выбран → «Выберите один товар»
- Override активен → нотификация

**Загрузка данных:**
- idle
- loading (спиннер/disabled кнопка)
- success (сообщение)
- error (текст ошибки)

## 7. Работа с WB API (обязательные методы)

Любой API-клиент для Wildberries должен уметь:

1. **Получить воронку продаж** — `POST /api/analytics/v3/sales-funnel/products`
   - Пагинация по 1000, курсор в `last_id`
   - Rate limit: при 429 — пауза ~21 секунда

2. **Получить поисковый отчёт** (опционально) — `POST /api/v2/search-report/report`
   - Требует тариф Jam
   - Если недоступен — soft error

3. **Получить детальный финансовый отчёт** — `POST /api/finance/v1/sales-reports/detailed`
   - Пагинация по rrdId (курсор)
   - Лимит 100000 строк
   - Поля: rrdId, nmId, vendorCode, title, subjectName, rrDate, saleDt, deliveryMethod, retailPriceWithDisc, ppvzSalesCommission, acquiringFee, rebillLogisticCost, paidStorage, penalty, deduction, additionalPayment, paidAcceptance

4. **Получить тарифы комиссий** — `GET /api/v1/tariffs/commission`

5. **Валидировать период:** from ≤ to, не старше 365 дней, не в будущем

## 8. Кэширование (как минимум на бэкенде)

Бэкенд должен кэшировать:
- Ответы юнит-экономики (ключ: api_key + date_from + date_to + product_id)
- Тарифы комиссий (ключ: api_key)
- Данные дашборда (ключ: api_key + date_from + date_to)

Кэш очищается при загрузке новых данных (Excel или WB API) и при сбросе.

## 9. Обработка ошибок WB API

| Код | Что делать |
|---|---|
| 401 | Сказать пользователю, что токен неверный |
| 429 | Подождать ~21 секунду, повторить (с логом) |
| 502/503 | Сказать, что WB временно недоступен |
| ConnectError / Timeout | Сказать, что не удалось подключиться к WB |
| Другое | Пробросить с оригинальным текстом ошибки |

## 10. Что делать не нужно (сознательно исключено)

- **База данных:** не нужна. In-memory хранилище.
- **Аутентификация/авторизация:** не нужна. Single-user, CORS открыт.
- **Рекомендации/советы:** не показывать. Пользователь сам принимает решения.
- **Сравнение произвольных периодов:** не делать (пока).
- **Экспорт CSV/PDF:** не делать (пока).
- **Ежедневная динамика (графики по дням):** не нужно.
- **Уведомления/алерты:** не нужно.

## 11. Общие принципы, которые стоит соблюдать

1. **Разделение состояния:** общие фильтры (general) → наследование → локальные (секции). override-флаги для каждой секции.
2. **Единый формат данных:** Excel и WB API → один и тот же processed_data.
3. **Все UI-компоненты имеют 4 состояния:** loading, empty, error, data.
4. **Логирование:** каждый важный шаг (загрузка, запрос к API, ошибка, сброс) должен быть в логе с префиксом (например `[UE]`, `[DASH]`, `[APPLY]`).
5. **Фронтенд не кэширует (пока)**, бэкенд кэширует.
6. **API-ключ WB** сохраняется в localStorage — чтобы не вводить при каждой загрузке страницы.
7. **Resolve ID** — обязательный шаг перед любым запросом к финансовому API WB.
8. **Rate limiting** — WB API строго лимитирует. Нужна глобальная блокировка (lock/asyncio.Lock) и пауза при 429.
